在這篇，我想要稍微練習一下，更加貼近真實應用的狀況

情境

當user使用一個有會員機制的網站時，一開始一定需要先註冊帳號

而當網站本身還要做權限控管的機制的話，那就必須要由一個相對權限高的人，來調整user的權限

而當user擁有應對的權限後，才能夠存取後端對應權限的資源

註冊帳號

一般註冊帳號，有的網站做的比較動態的，會在輸入帳號時，就即時的顯示該帳號是否有被用過

以及輸入密碼的同時，就同步顯示密碼強度

那這邊我們簡單起見，就還是交由後端驗證為主，並把驗證後的結果回傳到前端

開通帳號

在admin頁，允許admin開通帳號，而這個功能的實作有很多方式

而我這邊是用比較偷懶一點的方式去做

針對更新帳號的處理，會全部交由 api/Admin/UpdateAccounts 這支API來處理

並且API本身以接收多個帳號的方式來做處理

public IActionResult UpdateAccounts(IEnumerable<UpdateAccount> updateAccounts)

而開通帳號會是以點擊button的方式，來一筆一筆的呼叫 UpdateAccounts 這支API來更新資料

而更新資料時，由於更改帳號權限也是呼叫這一支API

因此UpdateAccount這個data model，刻意在各欄位上設計 isUpdatable 的欄位

用以讓後端直接判斷哪些欄位是會被更新的，這樣後端就不用再判斷要更新哪些資料了

public class UpdateAccount
{
    [Required]
    public int Id { get; set; }

    public string Name { get; set; }
    public bool IsUpdateName { get; set; }

    public AccountStatus AccountStatus { get; set; }
    public bool IsUpdateAccountStatus { get; set; }

    public ICollection<int> UserRoles { get; set; }
    public bool IsUpdateUserRoles { get; set; }

    }
}

更改帳號權限

而更改帳號權限這邊，則是採用比較類似表單傳送的方式做設計

並且用了一個防呆機制，透過isUpdateAccounts來判斷user是否有調整了表單

而判斷邏輯則是比對表單上的user role是否與原始資料相同

而當不同時，user一點擊update按鈕，就會呼叫 UpdateAccounts 這支API，並一次處理多筆資料

不過必須說的是，這樣的 UI 呈現方式，在資料量大的時候是非常不憂的

因為網頁上會直接顯示所有的 user 資料

實務上，還是會添加 pagination 的功能，一次要部分的資料

甚至搭配 filter，或search的功能，來讓 admin 更快地找到他想找的人

個人設定頁

個人設定頁就比較單純一點

一些資料是單純的顯示(Account Status、Email、Roles)

而 Name 則是做字串比對，跟原先的不同才允許更新，並且更新時，會連動更新 nav 上的顯示

而密碼的更新，則是做了double check，並且還要再輸入一次原先的密碼

總結來說

這次做了一些讓網站的一些設定值可以線上修改的功能，讓一些重要的參數不用進DB也可以修改

像是調整帳號是否已啟用，以及帳號可以擁有什麼權限，等等

但也因為做得比較簡單一點，因此user role是寫死的

一些參數前後端是寫死在code裡的

如果想要讓網站功能更加動態一點，這些參數寫進DB中

並由啟動階段來讀取他們，初始化網站，可能會是比較好一點的作法

此外在更新資料的機制上，其實這篇的做法就是資料庫的id前後端跑

這樣其實滿容易造成資安的風險

好一點的作法，應該是要將id加密後傳到前端，後端處理前再解密會好一點

另外，在網站的UI上，也比較陽春了一點

但若想要套用一些UI library，短期內可能會比較困難一些

畢竟vue3出的還不夠久，許多套件其實都還在規劃升級，或開發階段

只能說，vue3新推的功能是真的香

但若想要現在就使用vue3上營運環境
